## 项目结构与模块组织

Command Foundry 是一组独立维护的 CLI 工具。当前项目包括：

- `apps/codex-skills`
- `apps/ai-cli-manager`

每个应用的依赖、源码、测试、构建配置和文档应保留在各自的
`apps/<tool>/` 目录内。只有多个工具形成稳定的共同需求后，才在 `packages/`
中提取共享库。不要提交构建产物。

## 构建、测试与开发命令

进入目标应用目录后，按照该应用的 `README.md`、`package.json` 和锁文件安装依赖
并运行脚本。提交前至少执行该应用提供的测试和构建命令。Node.js 版本以应用的
`engines` 字段或 README 要求为准。

## 编码风格与命名

优先遵循目标应用的现有代码风格和工具配置。TypeScript 代码使用严格模式和 ES
模块；变量和函数使用 `camelCase`，类型使用 `PascalCase`，测试文件使用
`*.test.ts`。不要为局部改动引入无关的格式化或重构。

面向用户和贡献者的说明、帮助、错误提示、测试描述及必要注释优先使用简体中文；
命令参数、代码标识符和外部协议字段保持原文。

## 配置安全

修改用户配置时应保留无关内容，并采用适当的校验、并发保护和原子写入机制。
只读模式不得产生副作用，不得在测试中操作真实用户配置。
